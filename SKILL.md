# 人机协作 SDD 工作法 · 完整产品交付版

## 核心理念

**AI 负责结构和执行，人负责决策和把关。** 好的产品不是 MVP 迭代出来的，而是一次设计、一次交付、完整闭环。

SDD 把项目分成两个阶段：

```
阶段一：设计（Design）—— 规划完整业务闭环
  你用经验和直觉决策方向，AI 把决策落地为文档
  Constitution → Spec → Plan(含UI规范选择) → 你确认方案

阶段二：实现（Implement）—— 连续性执行直到交付
  AI 按计划逐层实现所有业务逻辑
  不暂停、不中断、不问你「下一步做什么」
  结束时交付完整可运行的产品
```

> **你只需要决策四次**：定方向 → 审方案 → 确认自审结果 → 验收交付。
> 中间的编码执行由 AI 全权完成，你无需在 Phase 间做确认。

## 反例黑名单

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|-------|------------|---------|
| 1 | "需求很清楚，直接开写" | 没有文档的范围会在实现中漂移 | 先走 Constitution → Spec → Plan |
| 2 | "先做核心功能，其他再说" | 部分功能做完了但业务流走不通，无法验收 | 一次设计完整业务闭环，确保每个流程可走通 |
| 3 | "改一个变量而已" | 变量改动可能隐含逻辑变更 | 更新 Spec 对应段落 |
| 4 | "这个很简单，不需要写文档" | 简单项目漏洞大多来自未明说的假设 | 轻量 Spec（3 个 Story）也要写 |
| 5 | "我先看看代码再回来确认" | 看了代码就会开始改，确认流程被跳过 | 先确认方案，后看代码 |
| 6 | "后面再补 Spec" | Spec 不写全，验收标准就不存在 | 先写 Spec，后实现 |
| 7 | "先跳过 Plan，直接写" | 没有 Plan 的代码没有结构 | Plan 至少列目录结构 + 数据流 |
| 8 | "测试后面再补" | 后面永远不会补 | 每个 Task 先写测试（RED→GREEN） |
| 9 | "我觉得这个调整很小" | 再小也要你确认，我说了不算 | 展示调整内容，ask_choice 让你决定 |
| 10 | "完成一部分先交付" | 业务逻辑没走通就交付无法验收 | 一次交付完整闭环，确保所有流程可验证 |

## 阶段一 · 设计（Design）

### Step 1: 定方向

我用 `ask_choice` 给出 3-5 个项目方向选项，你来选。也可以自定义。

关键决策：
- **做什么**（产品类型、目标用户、核心价值）
- **技术栈**（框架 / 数据库 / 部署方式）
- **业务闭环范围**（需要覆盖哪些完整业务流）

### Step 2: Constitution（项目宪法）

我起草 5-7 条核心原则 + 技术约束，你审阅。

**自审（我执行，不需你参与）：** 检查是否遗漏约束、有无矛盾条款、是否可落地。确认无误后展示给你。

- 批准 → 继续
- 修改 → 我调整后你再审

### Step 3: Spec（完整功能规格）

**目标：覆盖完整业务闭环。** 不是只写 P1 功能，而是把整个业务逻辑的所有环节都列出来。

我按**业务层**拆解，每层 3-8 个 Story，按 P1/P2/P3 排列：

| 业务层 | 举例（电商系统） | 覆盖内容 |
|--------|----------------|---------|
| **用户层** | 注册→登录→权限→认证 | 入口+身份 |
| **核心业务层** | 商品浏览→购物车→下单→支付 | 主流程闭环 |
| **支撑层** | 订单管理→退款→物流查询 | 售后+运营 |
| **管理层** | 后台管理→数据统计→系统设置 | 管理入口 |

每个 Story 包含：
- 用户场景描述
- 验收条件（Given/When/Then）
- 该功能在完整业务流中的位置（如：用户注册 → **下单** → 支付）

> **完整业务流的检验标准**：从一个用户进入系统开始，走完所有主要路径后退出，没有「此功能暂未实现」的断点。

**自审（我执行，不需你参与）：** 检查：是否有占位符？Story 之间是否有矛盾？验收条件是否可验证？各业务层之间能否衔接？是否存在「功能断层」（A→B→C 中 B 没写）？所有 Story 加起来能否构成一个可走通的业务闭环？修复后展示给你。

**文档保存：** 你确认 Spec 后，保存在 `docs/sdd/YYYY-MM-DD-spec.md`。

**你可以在这个阶段：**
- 增删功能
- 调整优先级
- 修改验收标准
- 重新排布业务层

### Step 4: Plan（完整技术方案）

我输出覆盖**所有业务层**的完整技术方案：

- **项目结构**（树形图，展示各业务层的目录映射）
- **业务层架构**（各层的模块职责、层间依赖关系）
- **数据模型**（所有实体及其关系，包含完整业务字段）
- **数据流**（完整请求链路，从入口到存储的完整路径）
- **API 设计**（所有接口：请求/响应/状态码/认证方式）
- **关键路径分析**（每个业务层的必经流程及其分支）
- **风险与边界条件**（并发、安全、容错限流方案）

#### Step 4a: 选择 UI 设计规范（前端项目时）

如果项目包含前端界面，在 Plan 阶段增加一道决策：

**我检查全局已安装的前端设计 skill（通过扫描 Skills Index 中 frontend/design/ui 相关的 inline skill），用 `ask_choice` 让你选择：**

| Skill | 风格定位 |
|-------|---------|
| `design-taste-frontend` | 反AI模板化前端设计 — Landing/Portfolio/Redesign 首选 |
| `minimalist-ui` | 极简编辑风（Notion/Linear 类似） |
| `high-end-visual-design` | 高级平静奢华视觉（品牌/奢侈品） |
| `industrial-brutalist-ui` | 粗野主义工业风格（创意/Awwwards） |
| `liquid-glass-ui` | 液态玻璃毛玻璃效果（Apple 风） |
| `frontend-design-gddw` | 前端设计 token 系统（色彩/字体/圆角/按钮） |
| `design-taste-frontend-v1` | v1 经典版，更偏创意激进的 GSAP + 物理动效 |
| `open-design-workflow` | 完整设计工作流（设计宪章+系统模板） |

**你也可以：**
- 自定义（描述你要的风格）
- 不选择（跳过，AI 用默认前端规范）
- 选择多个（如 `design-taste-frontend` 做主规范 + `liquid-glass-ui` 做毛玻璃卡片参考）

**选定后：** 所选 skill 的 `name:` 记录在 Plan 文档中。实现阶段该 skill 自动加载到上下文。

#### Step 4b: 技术方案输出

Plan 文档中额外包含：

- **UI 设计规范来源** — 所选 skill 的 name + 关键配置
- **组件架构** — 根据所选设计规范的风格偏好定制目录结构
- **主题系统** — CSS 变量策略（Tailwind dark: variant / CSS 变量 token 系统）

**自审（我执行，不需你参与）：** 检查：项目结构是否覆盖所有业务层？组件职责是否单一？数据模型是否支持所有 Story？API 是否完整？是否有遗漏的边界情况？所有类型签名和方法名在 Task 之间是否一致？所选 UI 设计规范是否已记录并准备好在实现阶段加载？修复后展示给你。

**文档保存：** 你确认 Plan 后，保存在 `docs/sdd/YYYY-MM-DD-plan.md`。

**STOP**：方案确认后进入阶段二。这是你最后一次把关。进入实现阶段后，除非发生重大变更，AI 将连续性执行直到交付。

## 阶段二 · 实现（Implement）

### 执行模式：连续性管道

```
                         你
                         │
                     [方向确认]
                    [方案确认]
                         │
                    ─────┘─────
                    ┌──────────┐
                    │ 实现管道  │
                    │ (Pipeline)│
                    └────┬─────┘
                         │  ┌───────────┐
                         ├──│ 业务层 1   │── 完成后自动推进
                         ├──│ 业务层 2   │── 完成后自动推进
                         ├──│ 业务层 3   │── 完成后自动推进
                         ├──│ 业务层 4   │── 完成后自动推进
                         ├──│ 集成测试   │── 整体回归
                         ├──│ 用户自审   │── 模拟走通+优化修复
                         └──│ 交付产物   │── 打包+构建验证
                         │
                    ─────┘─────
                         │
                     [验收交付]
```

**每个业务层的执行循环**（无需你介入）：

```
Task 1: 写测试（RED）→ 实现（GREEN）→ 重构 → 提交
Task 2: 写测试（RED）→ 实现（GREEN）→ 重构 → 提交
...
Task N: 写测试（RED）→ 实现（GREEN）→ 重构 → 提交

层内阶段：
1. 实现该层所有 Task
2. 运行该层全部测试，确认绿色
3. 按所选 UI 设计规范（如 Plan Step 4a 中选定）加载约束，检查界面代码是否符合规范
4. 按 code-quality 审视代码质量（命名/函数/错误/安全/反模式）
5. 记录任务状态
6. 自动进入下一业务层
```

**实现管道内置质量闸门**（不需要你操作，但自动执行）：

| 闸门 | 检查项 | 不通过则 |
|------|--------|---------|
| 测试门 | 该层所有测试通过 | 修复后继续 |
| 构建门 | 项目可编译/构建 | 修复后继续 |
| 质量门 | code-quality 清单过一遍 | 修复后继续 |
| UI门（前端项目） | 按所选 UI 设计规范检查视觉一致性 | 修复后继续 |
| 完整性门 | 该层所有 Story 满足验收条件 | 记录未完成项，继续下一层 |

**管道执行规则**：

1. 从底层向上层逐层实现（用户层 → 核心业务层 → 支撑层 → 管理层）
2. **后层可以使用前层的代码和数据模型**（已建立依赖）
3. **层间不暂停** — 前一层完成后自动进入下一层
4. **管道只停在这四种情况**：
   - **严重编译错误**（无法恢复）
   - **测试大面积失败**（超过 30% 测试不通过）
   - **你请求中断**（任何时候你都可以说「暂停」）
   - **管道完成所有业务层**（进入用户视角自审优化阶段）
5. 非上述情况，AI 自行修复后继续

### 首次构建后不重复重建

- 管道启动时创建完整项目脚手架
- **后续业务层只追加代码、不重建项目**
- 每次提交确保已有测试和新增测试都通过
- 完整的集成测试在**交付前**统一执行

---

### 用户视角自审与优化

管道执行完毕后、交付之前，AI 必须站在**最终用户的角度**审视产品，自行修复所有问题。

**自审流：**

1. **模拟用户走一遍主流程** —— 从头到尾操作，检查所有业务流能否走通
2. **检查边界体验** —— 错误提示是否友好？空状态有展示吗？加载中有反馈吗？
3. **检查异常流程** —— 输入错误数据、网络中断、权限不足时系统表现
4. **一致性检查** —— 命名风格、UI 风格、响应格式是否统一。如果 Plan 中选择了 UI 设计规范 skill，按该 skill 的 Pre-Flight Check 或质量清单过一遍视觉一致性
5. **性能安全嗅探** —— 有没有 N+1 查询？SQL 注入风险？重复请求？

**发现的问题，AI 自行修复闭环：**

- 不需要停下来问「这个要不要改」
- 修复后重新跑测试，确认绿色
- 修复后再回到自审流步骤 1 重新模拟一遍
- 直到自审清单全部通过，没有遗漏

> **完成条件**：你（作为最终用户）可以从头到尾走完所有业务流程，没有「研究中」、没有断点、没有不友好的提示。

### 交付产物

管道执行完毕后，自动生成以下交付物：

```
📦 交付包
├── 完整项目代码           ← 所有业务层已实现，项目可运行
├── docs/
│   ├── architecture.md    ← 最终架构（含实现中发生的调整）
│   ├── api-reference.md   ← 所有接口文档
│   └── changelog.md       ← 实现过程中的关键决策记录
├── 构建产物               ← 构建通过
└── 测试报告              ← 全部测试通过
```

**CHECKPOINT**：交付产物生成后，通知你来验收。你看代码、试运行、检查验收标准。通过则交付完成，不通过则回到设计阶段调整。

## 关键协作原则

1. **不跳过决策闸门** — 设计阶段每步确认后再推进，避免返工
2. **一次性设计，Solo 连续性执行** — 设计阶段充分讨论，实现阶段 AI 全权负责编码、测试、自审、优化、交付
3. **动态调整** — 任何时候你都可以提出修改，回到设计阶段更新后继续
4. **完整性先行** — 业务逻辑必须闭环，不允许「此功能暂未实现」的断点
5. **质量门自动拦截** — 测试、构建、代码质量、UI规范在管道内自动检查，不需要你参与
6. **可追溯** — Spec 和 Plan 确认后保存到 `docs/sdd/`，实现变更记录到 changelog

## 失败模式编码

### Step 1-4（设计阶段）

| 触发条件 | 症状 | 一线修复 | 仍失败兜底 |
|---------|------|---------|-----------|
| 用户对选项都不满意 | 无法推进到 Constitution | 让用户自定义方向 | 用最接近的选项先写草案，用户再改 |
| 选的技术栈不可行 | Plan 发现技术约束 | 在 Constitution 阶段调整 | 回退到 Step 1 重新选 |
| 用户连续 3 次修改仍不满意 | 同一阶段打转 | 暂停，问用户要不要缩小范围 | 缩减范围再重试 |
| 自审发现矛盾 | 条款/Story 之间冲突 | 标记冲突，提供修改建议 | 以用户最新指令为准 |
| 用户跳过审阅说"继续" | 未确认文档进入下一阶段 | 提示用户必须先确认 | 强制暂停，展示确认框 |
| 无可用的前端 UI skill | Plan 4a 找不到前端设计 skill | 扫描全局 skills 目录补充列表 | 直接让用户描述风格偏好 |

### Step 5-6（实现管道）

| 触发条件 | 症状 | 一线修复 | 仍失败兜底 |
|---------|------|---------|-----------|
| 业务层 Task 拆太大 | 一层超过 50 个 Task | 继续拆细到可原子提交 | 标记子层延期，先完成主流程 |
| 测试一直不通过 | 反复修改都失败 | 检查测试逻辑是否正确 | 向用户报告，一起看失败原因 |
| 构建验证失败 | 编译/测试/构建出错 | 查看错误日志，针对性修复 | 暂时移除出错的模块，标记 |
| 层间依赖不匹配 | 后层用了前层不存在的方法 | 检查 Plan 中的类型签名是否一致 | 修复后层适配前层实际接口 |
| UI 规范冲突 | 所选 skill 的规则与实际产出不匹配 | 加载 skill 重新检查不符合的组件 | 回退到 Step 4a 重新选择 |
| 用户请求中断 | 管道执行中被中断 | 保存当前进度（已完成的层+当前层完成部分） | 用户回来后恢复管道 |
| 管道严重错误 | 编译大面积失败，超过 3 次修复无效 | 暂停管道，向用户报告状态+建议方案 | 用户决策：回退/修改方案/继续硬修 |

## 适用场景

- **新项目启动（0→1）** — 从零构建完整产品
- **完整功能模块** — 为现有项目添加完整的独立业务模块（如：支付系统、用户系统）
